# Agent 层设计(Phase 4)

## 定位

单 Agent + ReAct 循环,面向个人知识库问答。不做多智能体:LiteSearch 是单一领域、本地 3B 模型驱动的场景,多智能体编排(planner/executor 分工、动态角色)解决的是长程复杂任务的协作问题,在这里属于过度设计。

设计上参考了 Aime(arXiv:2507.11988, ByteDance)中与单 Agent 场景相通的四个工程模式,并说明取舍。

## 借鉴的四个模式

### 1. Prompt 模块化组装

Aime 把 actor 的 system prompt 拆成 Persona + 工具描述 + 知识 + 环境 + 输出格式五个模块动态拼装(论文式 3)。本项目的 agent prompt 同样按模块组合,不硬编码一整块字符串:

- **Persona**:固定(知识库问答助手);
- **工具描述**:由注册的工具对象自动生成,新增工具不改 prompt 模板;
- **知识**:检索结果注入(这就是 RAG 上下文,对应论文的 κ);
- **输出格式**:严格的 JSON 结构约定(见下)。

对应 JD:"抽象通用开发框架与基础组件"。

### 2. 严格结构化输出 + 解析失败处理

论文强调输出必须按 schema 约束才能被可靠解析(式 3 的 Γ)。本项目用的 Qwen2.5-3B 格式遵循能力弱于大模型,ReAct 循环里 Action 解析失败预计是主要故障来源,因此:

- Agent 每轮输出约定为 JSON(`thought` / `action` / `action_input` / `final_answer`);
- 解析失败自动重试(带纠错提示),超过阈值则降级为直接检索;
- 每次解析失败记入 trace,作为 Phase 5 badcase 的一类固定来源。

### 3. 小工具集,精描述

论文的 Actor Factory 用工具 bundle 而非平铺全量工具,理由是全量工具会导致 LLM 选择低效或出错。3B 模型上该问题被放大,因此 v1 只提供 2-3 个工具:

- `search_notes`(必选):调用检索层;
- `calc_date`(辅助):日期计算。

每个工具带精确的一句话描述与参数 schema。

### 4. 执行过程实时进 trace

Aime 的 actor 通过 `Update_Progress` 在执行中途主动上报里程碑,而不是结束才汇报。映射到单 Agent:每轮 Thought/Action/Observation 都用现有的 `traced_stage()` 实时写入请求 trace,一次多轮工具调用的请求可以事后逐步复盘。

对应 JD:"日志、监控、故障定位"。

## 明确不做(v1)

- **Actor Factory / 动态实例化**:只有一种 actor,没有按任务组装的需求。

## 后续更新:「复杂需求」多 Agent 协作管线

上面"Dynamic Planner 留作 stretch"的判断后来被推翻了,原因不是补简历,而是两个真实理由:

1. **幻觉/编造风险一直没被系统性解决**——`/describe` 里"只能从候选花材中选,不要编造"只是一句 prompt 指令,没有任何校验,3B 模型完全可能不听。
2. **多收件人/多步骤需求出现了**("给妈妈和女朋友各配一束不同风格的花"),这正是当初搁置的 Planner 该接住的场景。

落地为一条新的 `/plan` 接口(`app/agent/harness.py` + `app/agent/pipeline.py`),Planner → Composer → Verifier 三个角色协作,和上面的单 Agent 检索问答是两条独立的能力,不互相替代。

### 核心设计:Verifier 拆成两级,只有确定性检查触发重试

- **花材清单核对(确定性,不调模型)**:Composer 除了"花束名""介绍",必须额外输出"花材清单"——文案里实际用到的花材名字。用 Python 把清单和这次候选花材的名字(从语料 `# 花名` 提取)逐个核对,把"有没有编造"从模糊问题变成机械检查。**候选集合是这次检索召回的那几种,不是全语料库**——所以哪怕某种花在语料库别处存在,只要没被召回给这次的 Composer,写出来就算不合规,这是刻意从严。只有这个检查失败才触发重写(最多 2 次尝试),仍失败则不再调模型,确定性地用候选里分数最高、且没在别的子任务用过的花材拼保底文案。
- **语气贴题复核(LLM judge,只跑一次不重试)**:花材核对通过后再判断文案是否贴合子任务场合。这是主观判断,3B 模型的判定本身不稳定,进重试循环可能永远收敛不了,所以只跑一次,不通过就记 badcase、前端标"评审建议",不阻塞返回。

实测中两种失败都真实出现过:候选里没给"洋桔梗""尤加利",模型仍凭训练知识写了出来,被机械核对逮到;一次"元气满满"的请求生成了偏"低谷安慰"语气的文案,被 judge 挑了出来——两级校验各自命中了不同类型的问题,证明拆开是对的。

### 跨子任务花材去重

多收件人场景的意义就是"风格不同",所以后面子任务的 Composer prompt 会带上"之前用过 XXX,换一批"的提示。**这个提示只在正常生成路径生效,保底方案一开始漏掉了**——保底直接抓分数最高的候选,不看是否已被别的子任务用过,实测就复现过两个子任务保底成同一种花。修复后保底逻辑改为跳过已用过的候选,优先选没用过的。

### 一个比预想更隐蔽的踩坑:contextvars 跨线程池 resumption 不传播

`/plan` 用 `StreamingResponse` 流式返回。最初实现里,`start_trace()`/`traced_stage()` 都在生成器内部按老方式直接调用,结果实测 trace 里**只记录了第一个 `yield` 之前的那一个阶段,后面所有 Agent 调用的耗时全部丢失**。

根因:FastAPI/Starlette 迭代一个普通同步生成器时(`iterate_in_threadpool`),每次 `next()` 调用都单独提交到线程池,而**每次提交都会 fork 一份独立的 contextvars context 副本**——第一次 `next()` 里 `start_trace()` 设置的 contextvar,不会传播到第二次 `next()` 的 context 副本里,`traced_stage()` 读到的是 `None`,静默不记录(`traced_stage` 设计上对 `trace is None` 是无操作而不是报错,所以这个问题不会以异常形式出现,只会表现为"数据莫名其妙不全")。

修复:把整条 pipeline 逻辑放进一个专门的后台线程,从 `start_trace()` 到最后一个 `yield` 全部在同一个线程、同一个 context 里跑完(线程内部执行不会被 Starlette 的分段调度打断),外层暴露给 `StreamingResponse` 的生成器只是阻塞读一个 `queue.Queue` 转发事件,不接触任何 contextvars。这个模式(pipeline 内的可观测性代码栈,和外层的流式调度分开)之后如果做更多流式接口可以直接复用。

### 已知局限(诚实记录)

- 最坏情况延迟约 30 秒(3 个子任务 × 最多 2 次生成 + 1 次 judge),流式只是让"等待"变得可见,不改变总耗时——这是模式的 demo,不是生产级延迟优化。
- `get_generation_model()` 是 `lru_cache` 单例包一个 MLX 模型实例,并发请求同时调 `.generate()` 没有加锁保护;这个功能因为单次请求占用 worker 更久,放大了这个已存在但未修的风险。
- badcase 日志(`backend/eval/badcases/badcases.jsonl`)目前只做写入,没有大小/轮转处理,也没有配套的看板——Phase 5 的读取/分析部分仍是待办。
